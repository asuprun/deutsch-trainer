# 02 — Архитектура

## Высокоуровневая диаграмма

```
┌────────────────────────────────────────────────────┐
│            PWA-клиент (Next.js, браузер)           │
│  ┌──────────────────────────────────────────────┐  │
│  │  Загрузка скрина / просмотр карт /           │  │
│  │  review-сессии (FSRS) / TTS / статистика     │  │
│  └──────────────────────────────────────────────┘  │
└─────────────┬──────────────────────┬───────────────┘
              │ HTTPS                │ HTTPS
              ▼                      ▼
┌──────────────────────┐   ┌─────────────────────────┐
│  Next.js API routes  │   │  Supabase                │
│  (server-side)       │──▶│  ├─ Postgres (карты)     │
│  /api/extract        │   │  └─ Storage (скрины)     │
│  /api/cards/*        │   └─────────────────────────┘
│  /api/review/*       │
│  /api/stats          │
└──────────┬───────────┘
           │ REST
           ▼
┌─────────────────────────────┐
│  Google Gemini 2.0 Flash    │
│  Vision API                  │
│  (структурированный JSON)    │
└─────────────────────────────┘
```

## Слои ответственности

### Клиент (браузер / PWA)
- Рендеринг UI, навигация.
- Локальная обработка изображения перед загрузкой (предварительное сжатие в WebP через canvas, если нужно).
- Воспроизведение TTS через Web Speech API (без бэкенда).
- Кэширование статики через service worker.
- Оптимистичные обновления — мгновенный отклик UI, откат при ошибке сервера.

### Next.js API routes (server-side)
- Всё взаимодействие с Gemini и Supabase идёт ТОЛЬКО через сервер.
- Никаких прямых вызовов из клиента к Gemini или к Supabase service-role.
- Password middleware защищает все маршруты, кроме `/login` и `/api/auth/*`.
- Финальное сжатие изображений через `sharp` (WebP, max 1600px, quality 85).
- FSRS-расчёты — на сервере (источник истины), клиент получает готовое состояние.

### Supabase
- Postgres хранит все данные приложения.
- Storage хранит оригиналы скринов (приватный bucket, доступ только через подписанные URL).
- RLS отключён — single-user, защита на уровне middleware Next.js.

### Gemini API
- Принимает изображение + системный промт + responseSchema.
- Возвращает валидный JSON со словами, грамматикой, примерами.
- Никогда не вызывается с клиента напрямую — ключ только на сервере.

## Ключевые потоки

### Поток 1 — Загрузка и извлечение

```
[Пользователь] → перетаскивает скрин на /upload
       │
       ▼
[Клиент] → сжимает в canvas до 1600px WebP (опционально, экономит трафик)
       │
       ▼
POST /api/extract (multipart/form-data)
       │
       ▼
[Сервер]
  1. Проверяет авторизацию (middleware)
  2. Считает sha256 изображения
  3. Проверяет cache в БД по hash — если уже извлекали, возвращает кешированный raw_extract
  4. Если новое: финальное сжатие через sharp
  5. Загружает в Supabase Storage → получает image_path
  6. Отправляет в Gemini с промтом + responseSchema
  7. Парсит JSON, валидирует через zod
  8. Возвращает клиенту preview-данные
       │
       ▼
[Клиент] → показывает ExtractPreview с чекбоксами для каждого слова/фразы/правила
       │
       ▼
[Пользователь] → отбирает нужное, нажимает "Сохранить"
       │
       ▼
POST /api/cards/bulk-create
       │
       ▼
[Сервер] → создаёт source + cards + grammar_notes в одной транзакции
       │
       ▼
[Клиент] → редирект на /cards с тостом "Добавлено 12 карт"
```

### Поток 2 — Тренировка

```
[Пользователь] → открывает /review
       │
       ▼
GET /api/review/queue?limit=20
       │
       ▼
[Сервер] → SELECT * FROM cards WHERE due_at <= now() ORDER BY due_at LIMIT 20
       │
       ▼
[Клиент] → загружает первую карту, показывает front
       │
       ▼
[Пользователь] → жмёт "Показать ответ" → видит back, примеры, аудио
       │
       ▼
[Пользователь] → жмёт одну из 4 кнопок rating (Again/Hard/Good/Easy)
       │
       ▼
POST /api/review/answer { card_id, rating }
       │
       ▼
[Сервер]
  1. SELECT текущее fsrs_state
  2. fsrs.next(card, now(), rating) → новое состояние
  3. UPDATE cards SET fsrs_state, due_at, reps++, lapses (если rating=1)
  4. INSERT INTO review_logs
  5. Возвращает next_due
       │
       ▼
[Клиент] → переход к следующей карте
```

### Поток 3 — Обогащение карты (генерация примеров)

```
[Пользователь] → на /cards/[id] жмёт "Сгенерировать примеры"
       │
       ▼
POST /api/cards/:id/enrich
       │
       ▼
[Сервер]
  1. Берёт front, word_type, gender карты
  2. Отправляет в Gemini промт enrich (см. 05-prompts)
  3. Получает JSON: примеры, мнемоника, синонимы
  4. UPDATE cards SET examples, mnemonic
       │
       ▼
[Клиент] → обновляет вид карты
```

## Решения по консистентности

- **Транзакции:** `bulk-create` атомарен — либо все карты + source создаются, либо ничего.
- **Дедупликация скринов:** по sha256 хешу изображения — нельзя дважды извлечь один и тот же файл.
- **Идемпотентность review/answer:** если запрос повторился (двойной клик), второй UPDATE даст тот же результат, потому что `fsrs.next` детерминирован при тех же входных данных и `now()` — но на всякий случай добавим debounce на клиенте.
- **Source of truth для FSRS:** сервер. Клиент никогда не считает next_due сам, только показывает.
